# Adelos — Verified Claims: Technical Architecture

The plain-language version is in the [overview](claims-overview.md). This document assumes familiarity with public-key crypto, HMAC, signatures, and the verifiable-credentials model.

---

## 1. What the dongle provides

The Adelos device is a hardware-bound, biometric-gated key holder. It has two distinct uses of its core HMAC primitive, `K` burned into the ESP32-C3 eFuse and never extractable:

**Session derivation** — for access and key-unwrapping:
```
HMAC(K, salt ‖ nonce) -> 32 bytes
```
The nonce is generated by the device, recorded as single-use, and released only after an on-device fingerprint match. The device never returns the same output twice — a hostile client cannot obtain a repeat response even for an identical salt.

**Stable derivation** — for keypairs and persistent pseudonyms:
```
HMAC(K, context) -> seed
```
Deterministic by design. The same context always produces the same key material, giving the holder a reproducible set of cryptographic identities without storing private keys. Fingerprint-gated; no nonce, because reproducibility is the point.

What the primitive gives you:

- **Possession proof** — the holder controls `K`.
- **Continuity** — the same `K` produces the same outputs over time.
- **Biometric binding** — each use requires the enrolled finger.
- **Hardware binding** — `K` cannot be copied off the device by software.

What it does not give you:

- **Attribute attestation.** The device cannot originate a trustworthy claim like "over 18." That requires an external issuer.
- **Unlinkability, automatically.** A naively reused key or directly-presented signature is a stable correlator. See §4.

## 2. The three roles

- **Issuer** — attests a claim about the holder's pseudonymous key. The issuer may know the real-world identity; that knowledge stays issuer-side.
- **Holder** — the user plus the Adelos device. Stores credentials encrypted, unwrapped only by dongle + fingerprint. Presents proofs.
- **Verifier** — holds a local keyring of trusted issuer public keys. Checks a presented proof against that keyring: trusted issuer signed this claim, holder demonstrates live control, claim is unrevoked. Returns a yes/no — no identity, no traceable key, no issuer contact.

The issuer is unavoidable. "Over 18" has to be vouched for by someone; the dongle binds and releases that vouch, it does not manufacture it.

## 3. How a presentation works

**Enrolment (once).**
The holder proves control of a pseudonymous key to the issuer. The issuer attests the predicate — e.g. *"controller of key `X` is an adult"* — and signs it. The signed credential is stored encrypted; the dongle and fingerprint are required to unwrap it.

**Presentation.**
The holder produces a zero-knowledge proof: *a trusted issuer (verifiable against the verifier's local keyring) signed an adulthood claim bound to a key I demonstrably control right now.* The proof is freshly randomised — two presentations are cryptographically unlinkable. The verifier checks the proof against its keyring and gets a binary result.

**What the verifier learns:**
- A trusted issuer asserts the predicate holds.
- The holder is physically present and in control of the device.

**What the verifier does not learn:**
- Which key the holder controls.
- Which issuer signed (the proof can hide this).
- Any attribute beyond the predicate presented.
- Anything that links this presentation to any other.

*In one line: this device confirms that a trusted issuer asserts the bearer is of age.*

## 4. Unlinkability

A naively issued credential — holder presents the same key `X` and the same issuer signature every time — is a stable tracking handle. Two mechanisms address this:

**Per-verifier pseudonyms (lighter, partial).**
Derive a stable key per verifier from the dongle: `X_v = f(HMAC(K, verifier_id))`. Each verifier sees a distinct key, breaking cross-verifier correlation. The verifier still sees a stable key within its own context, and the issuer cert chain must be structured to prove `X_v` descends from the root key without revealing it. Does not hide the issuer identity at presentation.

**Anonymous credentials (full unlinkability).**
Use a scheme designed for this — BBS+, CL signatures, or equivalent. The issuer issues a credential bound to a holder secret gated by the dongle. At presentation, the holder produces a zero-knowledge proof: "an issuer from the trusted set signed this claim bound to a key I control." The proof is freshly randomised each time — presentations do not correlate, and the issuer identity can be hidden from the verifier. No key is exposed.

Adelos's contribution in either case is the hardware-bound, biometric-gated, non-exportable holder secret — the secure element the credential scheme requires but does not itself supply.

## 5. Honest limits

- **Trust is not correctness.** The crypto proves the issuer *said* it, not that the issuer was *right*. Garbage attestation in, garbage trust out.
- **Issuer-side linkage persists.** The issuer knew `X` ↔ person at enrolment. Unlinkability is achieved toward verifiers. It is not achieved toward the issuer. A compelled or breached issuer can leak the original linkage.
- **Collusion re-identifies.** If issuer and verifier pool their data, the holder can be unmasked. The model assumes non-collusion or holder-chosen counterparties.
- **Revocation is open.** How a verifier learns a credential has been withdrawn — without a phone-home that reintroduces surveillance — is unsolved. Candidates: short-lived attestations with reissuance, cryptographic accumulators, privacy-preserving revocation lists. Keeping the keyring current is the verifier's operational concern.
- **No network-layer anonymity.** Adelos protects identity within the proof. IP/network anonymity is orthogonal.

## 6. Standards alignment

The issuer/holder/verifier split, selective disclosure, and predicate proofs map onto the W3C Verifiable Credentials data model and the anonymous-credential literature (BBS+ in particular). Where possible, Adelos should present as a standard holder/wallet with a hardware-bound, biometric-gated holder secret, rather than inventing wire formats — so existing issuers and verifiers can interoperate.

## 7. Summary

- The dongle gives possession, continuity, hardware binding, biometric gating.
- An external issuer supplies verified attributes — unavoidably.
- The verifier holds a local keyring; it is never contacted at presentation time.
- The presentation is a yes/no — no identity, no traceable key, no correlation between uses.
- Anonymous credentials make presentations cryptographically unlinkable.
- Adelos's contribution is the hardware-bound, biometric-gated, non-exportable, presence-enforced holder — the secure element for an anonymous-credential wallet, built FOSS for a few pounds.
